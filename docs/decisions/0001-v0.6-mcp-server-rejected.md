# ADR 0001: v0.6 MCP server proposed and rejected

- **Date:** 2026-05-28
- **Status:** Rejected (do not re-propose without addressing the prerequisites below)
- **Deciders:** Marcin Sufa (project owner), claude_code (spec author)
- **Reviewers:** `feature-dev:code-reviewer` subagent (verdict: REVISE-RECOMMENDED), `general-purpose` adversarial subagent (verdict: BLOCK), Marcin direct pushback

## Context

After shipping v0.5.1 (PR #6, squash commit `3b12499`, merged 2026-05-28), an external review by Composer 2.5 — pasted into chat during the v0.5.1 review iteration — suggested that v0.6 should extract dispatch logic from markdown `SKILL.md` into a dedicated `pr-autopilot-review` MCP server. The architectural argument:

- testable TypeScript adapters instead of LLM-interpreted markdown bash
- solve Gap D.2 (TRUE incremental progress) via polling
- channel abstraction for future adapters (Gemini, Codex CLI, others)

A full v0.6 spec was written (~700 lines, archived only in ExoVault memory `b2441212` since the worktree was force-removed during rejection cleanup) proposing:

- `pr-autopilot-review` MCP server (Node 20+ / TypeScript, stdio transport)
- 3 tools: `dispatch_spec_review(spec_path, channels?)`, `get_review_status(job_id)`, `list_review_findings(job_id)`
- File-based job state at `~/.pr-autopilot/review-jobs/<job-id>/`
- Channel adapters (TS): cursor-cloud-agent + codex-exec (Gemini deferred to v0.6.1)
- Backward compat fallback path preserving v0.5.1 dispatch when MCP not installed

## Decision

**REJECTED.** Do not ship v0.6 MCP server as currently specced. Defer indefinitely; revisit only when concrete v0.5.1+ production pain accumulates AND the prerequisites listed below are met.

The current ROADMAP slot for v0.7 ("Cursor-native runtime adapter — Path C") remains reserved. v0.5.2 was chosen as a smaller, more focused alternative (per-reviewer summary table + Gap E probe fix + this ADR).

## Reasoning

Three P0 findings from review iter1 (hostile reviewer's verdict was BLOCK):

### P0-1: Zero v0.5.1 production data justifies the rewrite

v0.5.1 shipped 2 hours before the v0.6 spec was written. No real-world incidents demonstrating the markdown-bash dispatch is broken in practice. The case rested on Composer's architectural elegance argument, not on empirical user pain. This is premature optimization risk: rewriting working code that has not yet been exercised against real workloads.

The "pain points" listed in the v0.6 spec ("not unit-testable," "polling logic in markdown," "adding a channel = SKILL.md edit") were theoretical, not observed.

### P0-2: D.2 incremental progress UX is fake in append-only Claude Code chat

The headline win of v0.6 was "live status table that updates mid-flight." But Claude Code chat is **append-only** — re-rendering a markdown table every 5s during a 120s poll loop produces **24 stacked status tables** in chat scrollback, not a live-updating UI. The user experience is worse than v0.5.1's atomic update at Step 4.

Mitigations considered:

- "Only re-render on state change" → cuts to ~3-5 messages, marginally better than atomic
- "Stream MCP responses" → requires Claude Code MCP streaming support that doesn't exist yet
- Neither achieves the intended UX without Claude Code platform changes outside our control

### P0-3: Post-Pro-upgrade reframe deflated the user case

The original v0.6 case leaned on Gemini Free adapter closing EVAL 42 ("no API keys") path for OSS / non-Pro users. After Marcin committed to Cursor Pro (validating EVAL 43 live the same day), the immediate user-value of MCP-dispatching to Gemini disappeared for Marcin's workflow.

What remained — "testable TypeScript adapters" — is solvable with ~4 hours of Bats tests on the existing bash dispatch, not a multi-day MCP rewrite. The cost/benefit ratio collapsed.

## P1 concerns (also reviewer-surfaced)

- `${CLAUDE_PLUGIN_ROOT}` likely doesn't expand in `.claude/settings.json` `mcpServers.args` array — load-bearing assumption unverified at spec time
- `claude mcp list` parsing for backward-compat detection is an unstable CLI contract
- Aggregation bridge between Claude subagent findings (in conversation context) and MCP-tracked findings (on disk) was undefined — could silently omit subagent findings from the final aggregate
- Backward compat doubles the maintenance surface: TS adapter shells out to existing bash probe = 3 layers of indirection for one curl call
- SKILL.md claimed to shrink ~30 lines; review showed it would grow ~30 lines once you count the MCP-availability detection wrapper + polling loop + fallback path

## Alternatives considered

1. **v0.5.2 per-reviewer summary table** (CHOSEN) — Marcin's actual user-value ask. Smaller scope. Same delivery shape as v0.5.1.
2. **Bats test suite on existing bash dispatch** — addresses the testability concern without MCP rewrite. ~4h work. Could ship if test pain materializes in production.
3. **Wait 4 weeks, gather v0.5.1 production data** — empirical justification first. Could revive the MCP idea with real incident evidence.
4. **graphify integration** (v0.5.3 candidate, memory `a9df909c`) — orthogonal token-cost reduction via knowledge-graph. Doesn't replace the MCP rewrite question but addresses a different pain.

## Consequences

- v0.5.2 ships as patch release (Gap E probe fix + per-reviewer summary table + this ADR + ROADMAP one-liner)
- v0.6 ROADMAP slot remains reserved for "Cursor-native runtime adapter (Path C)" — unaffected by this rejection
- ROADMAP Anti-roadmap section gains a one-line entry blocking re-proposal without prerequisite evidence
- ExoVault memory `b2441212` preserves the full rejection decision record for future reference

## Do not re-propose unless ALL of these prerequisites are demonstrated

A motivated re-proposer must furnish:

1. **N≥3 distinct user-reported incidents** in GitHub Issues naming markdown-bash dispatch as the cause. Single anecdotes don't count.
2. **Production deployment age ≥ 6 weeks** of v0.5.1+ with documented dispatch failures (not theoretical concerns from spec review).
3. **Working alternative attempted first** — specifically, a Bats test suite for the existing bash dispatch (per Alternatives §2 above). If Bats tests solve the testability concern, MCP is unnecessary.
4. **UX prototype in real Claude Code chat** showing how mid-flight updates render (NOT just text descriptions — screenshots of actual chat behavior across a 120s poll loop). The v0.6 D.2 design assumed live-updating UI but Claude Code chat is append-only — must prove the win is real, not 24 stacked tables.
5. **`${CLAUDE_PLUGIN_ROOT}` expansion verified** in `.claude/settings.json` `mcpServers.args` array, or an alternative resolution mechanism specified and documented.
6. **Aggregation bridge specified** between Claude subagent findings (in conversation context) and MCP-tracked findings (on disk). The v0.6 spec left this undefined; any revival must define it.

If ALL 6 prerequisites met → reopen as a v0.7+ proposal with this ADR linked from the new spec. The new spec must explicitly cite which incident(s) from prerequisite #1 motivate the change.
