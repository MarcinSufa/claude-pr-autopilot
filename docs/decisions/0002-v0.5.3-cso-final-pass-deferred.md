# ADR 0002: v0.5.3 `/cso` final-pass deferred pending upstream gstack `--non-interactive` flag

- **Date:** 2026-05-28
- **Status:** Deferred (do not implement without upstream prerequisites below)
- **Deciders:** Marcin Sufa (project owner), claude_code (spec author)
- **Reviewers:** `feature-dev:code-reviewer` subagent (verdict: REVISE — 2 P0 / 4 P1 / 3 P2), `general-purpose` adversarial subagent (verdict: REJECT — 5 P0 / 7 P1, score 1/5), `cursor-cloud-agent` (composer-2.5) timed out at 120s — pending
- **Spec under review:** [`docs/superpowers/specs/2026-05-28-pr-autopilot-v0.5.3-cso-final-pass.md`](../superpowers/specs/2026-05-28-pr-autopilot-v0.5.3-cso-final-pass.md) (kept as evidence)

## Context

After /cso (gstack's Chief Security Officer security audit skill) was suggested as a security gate for pr-autopilot, a v0.5.3 spec was drafted proposing `/cso` as an opt-in **final-pass reviewer** in Mode X step 9a. The integration plan:

- After primary reviewers (Cursor/Copilot/Codex) report green, invoke `/cso --diff origin/{base}..HEAD` via the Claude Code Skill mechanism (in-process, no external CLI).
- /cso writes its standard JSON report to `.gstack/security-reports/{date}-{HHMMSS}.json`.
- pr-autopilot parses the newest report, filters by `blockSeverity` (default `critical`), posts a summary PR comment, and PAUSEs the loop on blockers.
- Soft dep on gstack: pre-flight checks for `~/.claude/skills/cso/SKILL.md` existence; default `cso.enabled=false` so non-gstack users are unaffected.

Full spec ran to ~365 lines covering config schema, derivation table, pre-flight check, step 9a invocation, PR comment format, Mode Y exclusion, stop conditions, and EVAL scenarios.

## Decision

**DEFERRED.** Do not implement v0.5.3 as currently specced. The integration cannot ship safely until upstream gstack adds a `/cso --non-interactive` flag (or equivalent mechanism to suppress `AskUserQuestion` calls within the skill).

v0.5.3 is freed for an unrelated PATCH (e.g. graphify integration per memory `a9df909c`, or a different small improvement). Once gstack ships the prerequisite, this spec can be revived as v0.5.4 or v0.6.x.

## Reasoning

Two reviewers in `/pr-autopilot:review-spec --bootstrap` mode produced **7 P0 + 11 P1 + 3 P2 findings** in iter1. Two of the P0s are not "revise the spec" — they are **demonstrably false architectural assumptions** verified against the actual cso skill source:

### P0-1: "Skip Phase 13" loses the findings report itself

The spec's invocation block instructed Claude to "Run Phases 0-12 and 14, SKIP Phase 13 (Remediation Roadmap)". Verified against `~/.claude/skills/cso/SKILL.md` line 779:

> ### Phase 13: Findings Report + Trend Tracking + Remediation

Phase 13 contains the **Findings table** (line 783), **per-finding format** (line 796), **Incident Response Playbooks** (line 810), **Trend Tracking** (line 818), AND the Remediation Roadmap (line 834). Only the *last sub-section* (Remediation Roadmap) calls `AskUserQuestion`. Skipping all of Phase 13 means:

- No findings report generated.
- No `playbook` field per finding (skill JSON schema line 880 expects it).
- No `trend` block (skill JSON schema line 894 expects it).
- pr-autopilot's 9a.cso.3 parser breaks on the partial JSON, or worse, finds zero blockers in an incomplete report and lets bad PRs through.

The spec is **fundamentally wrong** on this point. Verified by grep against the live cso skill (v2.0.0, 927 lines).

### P0-2: Phase 8 ALSO calls `AskUserQuestion` — spec ignored it entirely

`~/.claude/skills/cso/SKILL.md` line 585:

> **Tier 2 — global skills (requires permission):** Before scanning globally installed skills or user settings, use AskUserQuestion

Phase 8 (Skill Supply Chain) runs in daily mode — it's not gated by `--diff` or `--comprehensive` flags. The spec only mitigates Phase 13. First cso-enabled PR run hangs at Phase 8 → AskUserQuestion → unattended loop never wakes. Found by adversarial reviewer; spec author missed it entirely.

### P0-3: No empirical proof the `AskUserQuestion` override mechanism works

Both reviewers independently flagged this. The spec's premise — "tell Claude in invocation instructions to skip Phase 13" — assumes prose instructions override hardcoded `AskUserQuestion` tool calls within the skill. There is no test, no precedent, no upstream gstack pattern confirming this works deterministically. Claude resolves the conflict by judgment, not by guaranteed behavior. If unreliable even 5% of the time, real PRs hang → manual intervention → trust in autopilot collapses.

The spec acknowledged this risk (§8 R1) but proposed shipping anyway. Reviewers rejected.

### P0-4: Newest-file glob is a security-bypass bug, not a correctness one

Spec §5.4 9a.cso.2 uses `newest_file(".gstack/security-reports/*.json")` + 600s age window to identify the report from "THIS invocation." Adversarial reviewer noted this creates a real failure mode:

- Two parallel `/loop /pr-autopilot:step` invocations on PR #100 and PR #101 (same repo).
- Both reach step 9a within the 10-minute window.
- Both glob; newest wins; loop A reads loop B's report.
- A's PR has a hardcoded secret. B's PR has none. A reads B's clean report → A auto-merges with the secret.
- **Security-bypass via race condition.** Worse than a correctness bug.

The spec already returns the absolute path from the skill invocation but **ignores it and globs anyway**. Trivial fix in design; missed in spec.

### P0-5: Default config has an audit-trail hole on fork PRs

Spec §9 Q3 left "Open" the question of what happens when `gh pr comment` fails (fork PRs, restricted bot permissions, rate limiting). Default `postPRComment=true` → comment fails → loop continues to safeAutoMerge without an audit trail. For a security-gating feature, silent merge on comment-post-failure is the wrong default. The spec shipped this hole as an open question instead of resolving it.

## P1 highlights

11 P1 findings; the most actionable:

- **Pre-flight renumbering scope undocumented.** Spec said "renumber 0.5 → 0.6, 0.6 → 0.7" but didn't enumerate the ~16 downstream references in skills/step/SKILL.md (lines 211, 264, 266, 271, 306, 612, 703, 714, 934, 981, 987, 1013, 1052-1055, 1062-1064, 1066). Better fix: insert as `0.4b` or named anchor, avoiding the renumber entirely.
- **`/cso --diff` leaks Phase 2/4 findings on files outside the PR diff.** Concrete UX disaster: typo-fix PR blocked on a year-old CI/CD finding in `.github/workflows/`. Users disable cso.enabled after one bad experience. Spec accepted this; reviewer rejected.
- **`cso` runs >1 time per loop** if cursor's score flaps 5→4→5. Each run costs 2-8min + Claude context. Spec hand-waved §6.6 "fires once near loop end" with no enforcement. Needs `state.csoLastReportHeadOid` caching.
- **`_finalPassOrder` constant location undefined.** Spec said "in code" but pr-autopilot is markdown skills, not code. Order needs explicit pseudocode in §5.4, not §6.5 prose.
- **Soft-dep detection by `~/.claude/skills/cso/SKILL.md` file existence is Windows-path fragile** + brittle to gstack install variations.

## Alternatives considered

1. **Defer + file gstack issue (CHOSEN).** File a feature request upstream for `/cso --non-interactive` mode that skips Phase 8 Tier-2 + Phase 13 Remediation Roadmap, leaving Phases 0-12 + the Phase 13 findings sub-section + Phase 14 (Save Report) intact. Minimal upstream surface. Unblocks v0.5.3 the moment gstack ships it. Cost: indefinite wait for gstack maintainer.

2. **Embedded rubric (no gstack dep).** Drop in-process skill invocation. Add a new `securityReview` final-pass reviewer that reads a `SECURITY-REVIEW-RUBRIC.md` checklist file in pr-autopilot (mirrors existing `claudeSelf` pattern). Loses gstack's CWE/OWASP taxonomy. Implementable now. **Rejected by Marcin (2026-05-28)** — gstack /cso is the source of truth; duplicating its rubric in pr-autopilot creates drift.

3. **Revise spec to address all 7 P0 + 11 P1 findings.** Stay on in-process /cso path. Rewrite §5.4 to skip only the Remediation Roadmap sub-section (not all of Phase 13); add Phase 8 mitigation (`--code` scope skips Phase 8 per cso skill line 345); fix race condition by trusting the path returned from skill invocation; decide gh-pr-comment fallback; enumerate renumbering refs; add 4 more EVAL scenarios. **Rejected** — even with all P0s fixed, P0-3 (no empirical proof the AskUserQuestion override works) remains unresolved without a real dogfood run, and there's no clean way to dogfood without ALSO touching upstream cso. Path 1 is strictly better.

4. **Stop entirely — security gating isn't pr-autopilot's concern.** Existing `claudeSelf` catches structural issues; users can run `/cso` manually outside the loop. **Rejected by Marcin** — security gate at PR-time is the whole point of the request.

- v0.5.3 release slot freed for an unrelated PATCH or skipped to v0.5.4 directly.

## Prerequisites to unblock (do not implement without ALL of these)

A future implementer must furnish:

1. **gstack ships `/cso --non-interactive` (or equivalent).** Definition: skips Phase 8 Tier-2 global-skills scan + Phase 13 Remediation Roadmap sub-section; runs everything else (Phases 0-12 + Phase 13 Findings Report sub-section + Phase 14 Save Report). Verified by checking the cso skill source at the version pr-autopilot will integrate against. Or alternatively: gstack ships a callable mode where pr-autopilot can pass the skill an explicit "auto-answer all AskUserQuestion with default" parameter.

2. **Empirical dogfood evidence.** Run the integrated `/cso --non-interactive --diff` against a real pr-autopilot PR (on this repo or a test repo) and observe:
   - Phase 8 does NOT halt the loop.
   - Phase 13 Findings Report IS generated.
   - Phase 14 Save Report writes the standard JSON.
   - The skill returns a path that pr-autopilot can read deterministically.
   - The whole thing completes in <10 min for daily mode.

3. **Race-condition fix specified.** Revised spec must use the skill's explicit return value (absolute path to `.gstack/security-reports/{date}-{HHMMSS}.json`), NOT a glob-newest heuristic. If the skill's invocation contract changes, the revised spec must call this out.

4. **PR-comment failure fallback decided.** Revised spec must define: what happens when `gh pr comment` returns non-zero. Recommend: retry-then-PAUSE on blocker runs (fail-close), graceful-degrade to PushNotification on clean runs (no audit trail loss because there's nothing to audit).

5. **Renumbering avoided OR enumerated.** Either insert the new pre-flight check as `0.4b` (no renumber), or list every downstream reference that needs touching in the SKILL.md.

6. **Composition order pseudocode in §5.4.** Explicit iteration order for final-pass reviewers (`claudeSelf → codex → copilot → cso`) baked into the SKILL.md pseudocode, not just discussed in §6.5 prose.

7. **EVAL scenarios cover timeout, blockSeverity=high, postPRComment=false, Mode Y ABORT.** Original spec only had 3 scenarios; reviewer P2 #7 listed 4 more required paths.

If ALL 7 prerequisites met → revive this spec as v0.5.4 or v0.6.x with a new spec file linking back to this ADR.
